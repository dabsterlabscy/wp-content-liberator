# wp-content-liberator

A **Claude skill** for taking a WordPress site whose content is trapped in hardcoded template files — and turning it into a real, editable, form-functional site **without paid plugins, SaaS subscriptions, or page builders**.

Use cases:

- You inherited a WordPress theme that was generated by a converter (WPConvert, create-react-app → WP, Vite → WP, hand-built build script) and the content is all in `.php` files
- You want to migrate off Elementor / Divi / Beaver Builder / Webflow but keep the site running
- Your developer handed you a theme that looks nice but has zero Customizer fields
- You need a contact form that hits your own webhook (n8n, CRM, custom endpoint) without a "forms-as-a-service" subscription

This skill encodes the patterns and specific bug fixes that let a single maintainer (or an AI assistant) do the liberation work predictably.

---

## What's a "Claude skill"?

A skill is a structured package that gives Claude (the AI assistant from Anthropic) persistent, specialized knowledge for a domain. When you install this skill into Claude, it will automatically consult the knowledge here whenever you're working on a WordPress theme with trapped content — no need to remind it. See [Anthropic's skills documentation](https://docs.claude.com/en/docs/build-with-claude/skills) for the mechanics.

You can also **read this repo as plain Markdown** without using Claude. The reference files and workflow are useful as a standalone field guide for developers.

## Installing as a Claude skill

Download the packaged `.skill` file from the latest release (or build it yourself — see below), then in Claude go to **Settings → Capabilities → Skills** and upload it.

### Building the `.skill` file yourself

Requires Anthropic's `skill-creator` skill available to package it. In Claude Code or Cowork:

```bash
cd /path/to/skill-creator
python3 -m scripts.package_skill /path/to/wp-content-liberator-repo /path/to/output/dir
```

This produces `wp-content-liberator.skill` in the output directory.

## Reading as a standalone guide

If you just want to learn the patterns without Claude involvement:

1. Start with **[SKILL.md](./SKILL.md)** — the overall workflow and three big gotchas
2. When you hit a cascade problem, go to **[references/bulletproof-overrides.md](./references/bulletproof-overrides.md)**
3. When you're ready to refactor templates, go to **[references/customizer-refactor.md](./references/customizer-refactor.md)**
4. When you need to wire forms, go to **[references/form-wiring.md](./references/form-wiring.md)**
5. When you hit a weird bug, check **[references/common-bugs.md](./references/common-bugs.md)** — the symptom catalog covers nine distinct bugs with proven fixes

The scripts folder has one utility: **[scripts/check_balance.py](./scripts/check_balance.py)** — run it before and after any edit to `functions.php`. Generated themes often have non-zero baseline brace balance; you need to preserve whatever baseline the file starts with.

## Scope

### Covered

- WordPress theme customization using only core features (Customizer, theme_mod, nav menus, hooks)
- Cascade conflicts in themes with multiple auto-enqueued stylesheets
- Hijacked native HTML (`<details>`, `<summary>`, `<button>`) by leftover builder JS
- Converting hardcoded markup into Customizer fields
- Form wiring to webhooks/n8n without SaaS dependencies
- Environment portability (staging ↔ production) with database search-replace workflow
- Mobile navigation patterns for themes that don't ship a working one
- Post-migration URL leakage diagnosis and fixes

### Not covered

- Building the original React/HTML source (that's upstream)
- WordPress core development, multisite, or server configuration
- Payment processing (beyond form-wiring to a downstream tool that handles it)
- Custom post types or complex taxonomy design
- Block editor / Gutenberg — this skill specifically addresses themes that store content in templates, not posts
- Performance tuning (use Cloudflare / WP Rocket / LiteSpeed separately)

## Background

This skill is the distilled output of a multi-week debugging and refactoring session on a real production site (an AI-automation consultancy targeting SMBs in Cyprus & Greece). The site was generated by WPConvert from a React source, and the process of making it actually maintainable surfaced many of the patterns here.

The specific bugs in `references/common-bugs.md` are not hypothetical — they're issues that actually happened, and the fixes shipped. If you're working on a WPConvert-generated site, you will almost certainly hit several of them.

## Contributing

Contributions welcome. Particularly useful additions:

- **More bug signatures** — if you've encountered a trapped-content issue not in `common-bugs.md`, document the symptom, root cause, and fix, and PR it.
- **Coverage for other generators** — the current skill is WPConvert-leaning. Patterns for handling output from other converters (`next-wp`, `gatsby-wp-export`, Vite + custom scripts, etc.) would broaden the skill's reach.
- **Test suite** — the skill-creator framework supports quantitative eval of skills. A test suite that exercises the triggering description and the workflow would catch regressions.

PRs that materially change the workflow should update `CHANGELOG.md` under an `## [Unreleased]` heading.

### How to add a new bug to common-bugs.md

Each entry should have:

1. A short numbered heading: `## N. Short symptom description`
2. A **Symptom** section — what the user would observe in a browser
3. A **Root cause** section — the technical explanation
4. A **Fix** section — the specific change, with code
5. Optional **Verification** section — how to confirm the fix worked

Follow the existing entries as examples.
